# 汽车双 ECU 仪表盘项目方案

## 1. 项目目标

基于现有开发板搭建一个模拟汽车“动力域 ECU + 显示域 ECU”的仪表盘项目：

- 动力域 ECU：采集/控制电机转速，周期性发送转速、电机状态、故障状态等数据。
- 显示域 ECU：运行 bootloader、LVGL 仪表盘 UI，通过 CAN 接收动力域数据并显示。
- OTA/IAP：实现显示域应用升级，后续可扩展动力域远程升级。
- 通信链路：优先使用 CAN 总线，贴近真实车载 ECU 架构。

推荐主线：`STM32F407ZGT6 显示域 + STM32F103 动力域 + CAN 总线`。

独立备选：`ESP32-DOWN-V3 显示/网关方案`，用于 WiFi OTA 或作为后续联网扩展。

## 2. 已有硬件资源评估

### 2.1 STM32F407ZGT6 开发板

已知条件：优信电子购买，板上有排针、两个用户按键、两个 LED、Type-C 口等；当前没有 SD 卡，但手上有 AT24C02 EEPROM 和 W25Q64 SPI Flash。

推荐角色：显示域 ECU。

适合原因：

- 主频 168 MHz，运行 LVGL 足够。
- 片上 Flash 通常为 1 MB，适合做 bootloader + app 分区。
- SRAM 192 KB，不能做全屏双缓冲，但可以做 LVGL 局部刷新缓冲。
- 带 FSMC/FMC 类并口能力，适合驱动 8080 并口 TFT 屏。
- 带 bxCAN，适合模拟车载 CAN 通信。
- 可外接 W25Q64 作为固件暂存区、资源区和日志区。

外部存储建议分工：

- W25Q64：64 Mbit，也就是 8 MB，适合存放待升级固件、UI 图片/字体资源、运行日志。
- AT24C02：2 Kbit，也就是 256 Byte，只适合存少量参数、升级标志、版本号、CRC、启动计数等，不适合存固件。

注意点：

- 如果开发板没有外扩 SDRAM，不建议上过高分辨率屏幕。
- LVGL 要使用小块 draw buffer，例如 480 x 40 或 480 x 60，而不是整屏 framebuffer。
- 屏幕接口优先选 FSMC/8080 并口，不建议大尺寸 SPI 屏作为主仪表盘。

### 2.2 STM32F103 核心板

推荐角色：动力域 ECU。

适合原因：

- TIM 定时器适合 PWM 输出、电机测速、编码器计数。
- 带 bxCAN，能直接接 CAN 收发器。
- 性能足够处理直流电机闭环转速控制、周期报文发送、故障检测。

### 2.3 ESP32-DOWN-V3

推荐角色：备选显示域 / WiFi OTA 网关 / 后续联网模块。

适合原因：

- 双核 240 MHz，带 WiFi、BLE。
- 可做 Web OTA、HTTP 固件下载、手机/PC 网页配置。
- 有 TWAI 控制器，可通过外部 CAN 收发器接入 CAN 总线。

限制：

- 作为“显示域 + LVGL + 并口大屏”的体验通常不如 F407 直观，具体取决于屏幕接口。
- ESP32-DOWN-V3 若只是普通 WROOM-32 开发板，屏幕排线和并口扩展没有 F407 方便。

### 2.4 ESP32-C3

推荐角色：暂不作为主控，后续可做 WiFi 辅助模块。

原因：

- 资源够做联网，但不适合当前项目的主显示域。
- 一般无原生 CAN/TWAI 或板级支持不如 ESP32-DOWN-V3 直接。

### 2.5 现有 2208 无刷电机套件

当前结构：ESP32 插在专属电机驱动板上，驱动板连接 2208 三相电机和底部 AS5600 磁编码器。

结论：不建议作为本项目主动力域电机。

原因：

- 套件结构较固定，F103 很难自然接管驱动链路。
- 2208 无刷电机需要三相逆变桥、FOC 或六步换相，开发复杂度明显增加。
- 当前项目重点是双 ECU 架构、CAN、LVGL、bootloader/OTA，不宜把主要风险压在 BLDC 驱动上。

建议：

- 保留 2208 套件作为后续“进阶动力域/FOC 版本”。
- 第一版改用带编码器的有刷直流减速电机，更利于快速完成闭环控制和仪表盘联调。

## 3. 推荐采购清单

### 3.1 显示屏选型

首选：`3.5 英寸 480x320 TFT，8080 16 位并口，ILI9486/ILI9488 驱动，带触摸可选`

推荐理由：

- 尺寸接近小型汽车仪表/中控小屏，视觉效果比 2.8 英寸好。
- 480x320 足够做速度表、转速表、状态栏、故障灯、OTA 页面。
- 8080 并口配合 F407 的 FSMC/FMC 类接口，刷新率明显优于 SPI。
- LVGL 局部刷新可控，F407 片上 RAM 也能承受。

可接受低成本方案：`2.8 英寸 320x240 TFT，8080 并口或 SPI，ILI9341 驱动`

- 优点：便宜、资料多、调试简单。
- 缺点：屏幕偏小，仪表盘观感弱一些。

不建议第一版选择：`4.3 英寸 800x480 RGB 屏`

- 原因：通常需要 RGB 接口、较大 framebuffer、外扩 SDRAM 或 LTDC 资源配合。
- F407ZGT6 理论上可以折腾，但第一版复杂度偏高。

购买关键词：

- “3.5 寸 TFT 480x320 8080 并口 ILI9488 STM32”
- “3.5 寸 TFT 480x320 FSMC ILI9486”
- 优先选带原理图、示例代码、引脚定义清楚的模块。

触摸建议：

- 第一版可以不接触摸，只用两个用户按键切换页面/触发 OTA。
- 如果屏幕自带 XPT2046 电阻触摸，可以作为第二阶段接入。

### 3.2 CAN 收发器

已有：TJA1050 模块 1 块。

还需要：再购买 1 块 CAN 收发器模块。

推荐：

- 若 F407/F103 都是 5V 供电兼容环境：继续买 `TJA1050`，最简单。
- 若想兼顾 3.3V 逻辑与更稳妥电平：买 `SN65HVD230` 模块。

注意：

- TJA1050 通常是 5V CAN 收发器，MCU TX/RX 逻辑兼容性要看模块设计和开发板引脚容忍度。
- CANH/CANL 两端建议预留 120 欧终端电阻。短线桌面实验可以先只在总线两端各放一个 120 欧，或者模块自带跳帽就按需要接入。

### 3.3 电机与驱动模块选型

第一版推荐：`带霍尔编码器的有刷直流减速电机 + H 桥驱动模块`

#### 推荐电机 1：JGA25-370 带霍尔编码器

适合第一版。

建议规格：

- 额定电压：12 V
- 空载转速：100 到 300 RPM
- 编码器：AB 相霍尔编码器
- 减速箱：金属齿轮优先

优点：

- 体积小，价格低，资料多。
- 转速范围适合做仪表盘指针/数字速度显示。
- F103 可以直接采集 AB 相编码器或捕获脉冲测速。

#### 推荐电机 2：JGB37-520 带霍尔编码器

适合想要更稳、更有负载感的版本。

建议规格：

- 额定电压：12 V
- 空载转速：100 到 200 RPM
- 编码器：AB 相霍尔编码器

优点：

- 扭矩更大，机械质感更好。
- 堵转电流相对更高，需要更可靠驱动模块。

#### 不建议第一版购买

- 裸三相无刷电机：需要 FOC/六步换相和三相驱动，复杂度太高。
- 普通无编码器直流电机：不能做真实测速闭环，只能估算。
- 步进电机：能控制位置，但不太适合模拟汽车动力域转速反馈。

### 3.4 电机驱动模块

优先推荐：`BTS7960/IBT-2 H 桥驱动模块`

适合原因：

- 比 TB6612 更抗堵转和误操作。
- 适合 12 V 有刷直流电机。
- 控制简单：PWM + 方向，或双 PWM。
- 桌面实验余量大，不容易一上电就烧。

次推荐：`DRV8871/DRV8876/DRV8251A 模块`

适合原因：

- 体积小，效率高，适合小电机。
- 比 TB6612 更现代，保护特性通常更好。

谨慎使用：`TB6612`

- 小电机、低负载可以用。
- 堵转、电源反接、浪涌时容易损坏，不建议作为第一版主驱动。

不推荐：`L298N`

- 压降低、发热大、效率差，虽然便宜但体验一般。

## 4. 主方案系统架构

```text
                 CAN 总线
        CANH/CANL + 120R 终端

+----------------------+          +----------------------+
| 显示域 ECU            |          | 动力域 ECU            |
| STM32F407ZGT6         |          | STM32F103             |
|                      |          |                      |
| Bootloader/IAP       |          | 电机控制任务           |
| LVGL 仪表盘 UI        |          | 编码器测速             |
| CAN 接收与诊断        | <------> | PID/开环 PWM 控制      |
| W25Q64 固件暂存       |          | CAN 周期报文发送       |
| AT24C02 参数/标志     |          | 故障检测               |
| 用户按键/LED          |          |                      |
+----------+-----------+          +----------+-----------+
           |                                 |
           | 8080 并口                        | PWM/方向/编码器

      3.5 寸 TFT 屏                    DC 减速电机 + H 桥
```

## 5. 软件模块拆分

### 5.1 显示域 STM32F407

建议工程拆分：

```text
display_ecu_f407/
  bootloader/                # Bootloader/IAP 工程
  app/                       # LVGL 应用工程
  common/
    protocol/                # CAN 报文定义、数据结构、CRC
    bsp/                     # 板级公共定义
```

Bootloader 功能：

- 上电检查 App 有效性。
- 支持从 W25Q64 读取待升级固件镜像。
- 校验固件 CRC32。
- 写入 App 分区。
- 跳转到 App。
- 使用 AT24C02 或片内 Flash 参数区保存升级标志、版本号、CRC 和回滚状态。

App 功能：

- LVGL 初始化与页面管理。
- CAN 接收动力域数据。
- 显示转速、目标转速、电机状态、CAN 通信状态、故障码。
- 用户按键切换页面：主仪表页、诊断页、OTA 页。
- 可选：将关键运行状态写入 W25Q64 日志区。

推荐 Flash 分区：

```text
0x08000000 - 0x08007FFF   Bootloader    32 KB
0x08008000 - 0x0807FFFF   App 主区      480 KB 左右
0x08080000 - 0x080F7FFF   App 备份区    480 KB 左右，可第二阶段再做
0x080F8000 - 0x080FFFFF   参数/标志区   32 KB
```

第一阶段可以先做：Bootloader + 单 App 区 + 参数区。

第二阶段再扩展：A/B 分区、回滚、版本管理。

推荐 W25Q64 外部 Flash 分区：

```text
0x000000 - 0x0FFFFF   固件暂存区 A    1 MB
0x100000 - 0x1FFFFF   固件暂存区 B    1 MB，可第二阶段再用
0x200000 - 0x3FFFFF   LVGL 图片/字体资源区
0x400000 - 0x7DFFFF   日志/诊断数据区
0x7E0000 - 0x7FFFFF   外部 Flash 元数据/备份参数
```

推荐 AT24C02 数据规划：

```text
0x00 - 0x03   magic
0x04 - 0x07   boot_count
0x08 - 0x0B   current_version
0x0C - 0x0F   pending_version
0x10 - 0x13   pending_crc32
0x14 - 0x17   pending_size
0x18          upgrade_request
0x19          last_upgrade_result
0x1A          rollback_request
0x1B - 0x1F   reserved
```

### 5.2 动力域 STM32F103

建议工程拆分：

```text
power_ecu_f103/
  app/
    motor/                   # 电机 PWM、方向、使能
    encoder/                 # 编码器测速
    control/                 # 开环/PID 控制
    can_app/                 # CAN 报文收发
    diag/                    # 故障检测
    protocol/                # 与显示域共享的协议定义
```

功能：

- PWM 控制电机转速。
- 采集 AB 相霍尔编码器。
- 计算实时 RPM。
- 接收显示域命令，例如目标转速、启停、清故障。
- 周期性发送状态帧。
- 检测堵转、过流输入、编码器丢脉冲、CAN 超时等故障。

第一版控制策略：

- 先做开环 PWM：按键或 CAN 命令设置占空比。
- 再做速度闭环 PID：目标 RPM 与实际 RPM 闭环。
- 最后做故障降级：堵转/编码器异常时停机并上报故障。

## 6. CAN 通信协议草案

建议波特率：`500 kbps`。

桌面短线调试也可用 `250 kbps`，抗干扰余量更大。

### 6.1 动力域到显示域

#### 0x180 Motor_Status，周期 20 ms

| Byte | 内容 | 说明 |
| --- | --- | --- |
| 0-1 | actual_rpm | uint16，小端，单位 rpm |
| 2-3 | target_rpm | uint16，小端，单位 rpm |
| 4 | pwm_percent | 0-100 |
| 5 | motor_state | 0 stop, 1 run, 2 fault |
| 6 | fault_code | 0 表示无故障 |
| 7 | alive_counter | 0-255 循环 |

#### 0x181 Motor_Diag，周期 100 ms

| Byte | 内容 | 说明 |
| --- | --- | --- |
| 0 | temperature | 可先填 0xFF，后续扩展 |
| 1 | voltage_x10 | 电源电压 x10，可选 |
| 2 | current_x10 | 电流 x10，可选 |
| 3 | control_mode | 0 open-loop, 1 speed-loop |
| 4-7 | reserved | 预留 |

### 6.2 显示域到动力域

#### 0x200 Motor_Command，按需或 50 ms 周期

| Byte | 内容 | 说明 |
| --- | --- | --- |
| 0 | enable | 0 stop, 1 run |
| 1 | mode | 0 open-loop, 1 speed-loop |
| 2-3 | target_rpm | uint16，小端 |
| 4 | target_pwm | 0-100，开环模式使用 |
| 5 | clear_fault | 1 表示清故障 |
| 6 | command_counter | 0-255 循环 |
| 7 | checksum | 简单累加校验或预留 |

## 7. LVGL 仪表盘页面规划

### 7.1 主仪表页

显示内容：

- 大号速度/转速数字。
- 圆弧仪表盘或横向进度条。
- 电机状态：停止、运行、故障。
- CAN 状态：在线、超时。
- OTA/W25Q64 状态小图标。

### 7.2 诊断页

显示内容：

- actual_rpm、target_rpm、pwm_percent。
- CAN alive counter。
- fault_code 文本解释。
- 最近一次报文时间。

### 7.3 OTA 页

显示内容：

- 当前 App 版本。
- W25Q64 暂存区中检测到的固件版本。
- 升级进度条。
- 校验结果。
- 升级成功/失败状态。

交互建议：

- 用户按键 1：切换页面。
- 用户按键 2：短按切换启停，长按进入 OTA 确认。
- LED 1：系统心跳。
- LED 2：CAN 在线/故障状态。

## 8. OTA/IAP 实现路线

### 8.1 第一版：串口/YMODEM + W25Q64 本地升级

流程：

```text
1. PC 编译生成 display_app.bin
2. F407 App 或 Bootloader 进入升级接收模式
3. PC 通过串口/YMODEM 发送 display_app.bin
4. F407 将固件先写入 W25Q64 固件暂存区
5. 写入完成后计算 CRC32，并把 size/version/crc/upgrade_request 写入 AT24C02
6. 复位进入 Bootloader
7. Bootloader 读取 AT24C02 升级请求
8. Bootloader 从 W25Q64 读取固件并校验 CRC32
9. 擦除片内 App 区并写入新固件
10. 更新 AT24C02 升级结果标志
11. 跳转 App
```

优点：

- 不依赖网络，最稳定。
- 不依赖 SD 卡，完全利用你已有的 W25Q64 和 AT24C02。
- W25Q64 容量 8 MB，足够暂存多个 F407 App 固件。
- AT24C02 容量虽小，但很适合保存升级状态和少量参数。

注意：

- AT24C02 只有 256 Byte，不要把日志、UI 资源或固件放进去。
- W25Q64 通过 SPI 连接，建议使用独立片选，SPI 速率先保守设置，跑通后再提速。
- Bootloader 里要尽量少依赖复杂库，只保留 SPI Flash、EEPROM、CRC、Flash 擦写、串口日志等必要模块。

### 8.2 第二版：App 接收固件 + Bootloader 搬运

适合改善使用体验。

流程：

- 正常 App 运行时进入 OTA 页面。
- App 通过串口、USB 虚拟串口或 ESP32 网关接收固件。
- App 将固件写入 W25Q64 暂存区。
- App 写 AT24C02 的升级请求标志。
- 系统复位后 Bootloader 完成校验和搬运。

优点：

- 用户不需要每次手动进入 Bootloader。
- LVGL 可以显示接收进度、校验结果和错误原因。
- 这一版已经接近真实 ECU 的“应用层触发升级，bootloader 执行刷写”流程。

### 8.3 第三版：ESP32 网关 WiFi OTA

适合体现“远程 OTA”。

流程：

- ESP32-DOWN-V3 连接 WiFi。
- 从 HTTP 服务器下载固件。
- 通过 UART/CAN 将固件分包发送给 F407 App 或 Bootloader。
- F407 先写入 W25Q64 暂存区。
- Bootloader 校验并写入片内 App 分区。

建议第三阶段再做，不建议第一版直接上远程 OTA。

## 9. ESP32-DOWN-V3 独立备选方案

### 9.1 备选方案 A：ESP32 作为 WiFi OTA 网关

这是最推荐的 ESP32 备选用法。

架构：

```text
手机/PC/HTTP服务器
        |
       WiFi
        |
 ESP32-DOWN-V3 OTA 网关
        |
   UART 或 CAN
        |
 STM32F407 显示域 Bootloader
```

职责：

- ESP32 负责联网、下载固件、提供网页配置界面。
- F407 仍负责 LVGL 显示和最终 Flash 写入。
- 动力域 F103 不变。

优点：

- 保留 F407 做显示域的稳定性。
- ESP32 的 WiFi 优势被充分利用。
- 后续可扩展网页仪表盘、日志上传、手机控制。

缺点：

- 多一块板，供电和通信链路更复杂。

### 9.2 备选方案 B：ESP32 作为显示域主控

架构：

```text
ESP32-DOWN-V3 + 小尺寸 SPI TFT + LVGL
        |
      TWAI/CAN
        |
STM32F103 动力域
```

适用条件：

- 购买 SPI 屏，例如 2.4/2.8 寸 ILI9341。
- 接受 UI 刷新率和屏幕尺寸低于 F407 并口屏方案。

优点：

- WiFi OTA 很方便。
- ESP-IDF 生态完整。

缺点：

- 作为汽车仪表盘显示效果不如 F407 + 3.5 寸并口屏。
- 若要接 CAN，还需要外部 CAN 收发器。
- SPI 屏刷新大面积 LVGL 动画会吃力。

结论：

- 第一主线仍然使用 F407 做显示域。
- ESP32-DOWN-V3 更适合在第二/第三阶段作为 WiFi OTA 网关加入。

## 10. 开发阶段计划

### Phase 0：硬件采购与连线验证

目标：买齐最小闭环硬件，确认每个模块单独可用。

清单：

- 3.5 寸 480x320 8080 并口 TFT 屏。
- CAN 收发器模块再买 1 块。
- JGA25-370 或 JGB37-520 带霍尔编码器直流减速电机。
- BTS7960/IBT-2 或 DRV8871/DRV8876 驱动模块。
- 12 V 电源，建议 2 A 以上。
- 杜邦线、120 欧电阻若干。

验收：

- F407 点亮屏幕。
- F103 输出 PWM，电机可调速。
- F103 读取编码器脉冲。
- F407 与 F103 CAN 收发正常。

### Phase 1：动力域最小系统

目标：F103 独立完成电机运行和测速。

任务：

- PWM 控制电机正转/停止。
- 编码器测速，计算 RPM。
- 串口打印 RPM、PWM、状态。
- 加入简单开环控制。

验收：

- 串口能稳定输出实时 RPM。
- 改变 PWM 后 RPM 有对应变化。

### Phase 2：CAN 通信闭环

目标：F103 与 F407 通过 CAN 交换数据。

任务：

- F103 发送 0x180 Motor_Status。
- F407 接收并解析。
- F407 发送 0x200 Motor_Command。
- F103 根据命令启停或调整目标转速。

验收：

- 断开 CAN 时 F407 能显示超时。
- 恢复 CAN 后自动显示在线。
- F407 可以控制 F103 电机启停。

### Phase 3：LVGL 仪表盘

目标：F407 显示完整仪表盘 UI。

任务：

- 移植 LCD 驱动和 LVGL。
- 做主仪表页、诊断页、OTA 页。
- 按键切换页面。
- CAN 数据驱动 UI 刷新。

验收：

- 转速显示实时变化。
- CAN 超时、故障状态有明显提示。
- UI 刷新流畅，无明显撕裂或卡死。

### Phase 4：Bootloader/IAP

目标：F407 支持 W25Q64 暂存固件、AT24C02 保存升级状态，并完成 App 升级。

任务：

- 编写 bootloader。
- 实现 App 跳转。
- 实现 Flash 擦写。
- 实现 W25Q64 SPI Flash 读写。
- 实现 AT24C02 参数读写。
- 实现串口/YMODEM 接收固件到 W25Q64。
- 实现 CRC32 校验。
- App 显示版本号。

验收：

- 通过串口发送新 bin 后可升级。
- 升级失败不跳转无效 App。
- 断电或校验失败后不会破坏旧 App。
- App 版本号更新正确。

### Phase 5：闭环控制与诊断增强

目标：让项目更像真实 ECU。

任务：

- F103 增加速度 PID。
- 增加堵转/编码器丢失/CAN 超时诊断。
- 增加故障码表。
- F407 诊断页显示故障详情。

验收：

- 目标转速变化后实际转速可跟随。
- 故障触发后动力域停机并上报。
- 显示域能正确显示故障码。

### Phase 6：ESP32 WiFi OTA 网关

目标：把 ESP32-DOWN-V3 加入系统。

任务：

- ESP32 连接 WiFi。
- 提供网页上传固件或 HTTP 下载固件。
- ESP32 与 F407 通过 UART/CAN 传输固件分包。
- F407 bootloader 接收并写入。

验收：

- PC/手机可通过网页触发升级。
- F407 能完成远程固件更新。

## 12. 当前关键决策结论

1. 主控架构：采用 F407 显示域 + F103 动力域。
2. 通信：采用 CAN，不建议第一版用 UART 替代。
3. 屏幕：推荐 3.5 寸 480x320 8080 并口 TFT。
4. 电机：不使用现有 2208 无刷套件作为第一版主对象，改买带霍尔编码器的有刷直流减速电机。
5. 电机驱动：推荐 BTS7960/IBT-2，避免 TB6612 作为主驱动。
6. OTA：第一版做 F407 的 `串口/YMODEM + W25Q64 暂存 + AT24C02 标志` IAP，后续用 ESP32-DOWN-V3 扩展 WiFi OTA。

## 13. 下一步建议

下一步最适合先做两件事：

1. 确认 F407 开发板具体原理图或引脚表，重点确认 FSMC/屏幕接口、CAN 引脚、SPI 引脚、I2C 引脚。
2. 根据屏幕模块资料，确定 F407 到 TFT 的具体接线，然后开始建立 `display_ecu_f407` 和 `power_ecu_f103` 两个工程。

如果暂时没有原理图，也可以先按模块单独验证：先跑通 F103 电机测速，再跑通 F407 CAN 接收，最后移植屏幕和 LVGL。
